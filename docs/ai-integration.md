# AI Integration — Plan Generation

## Overview

Training plans are generated by calling an OpenAI-compatible chat completions API (OpenRouter by default) from the `createPlan` server action (`app/src/app/actions.ts`). No Anthropic SDK is used — the integration is a plain `fetch` call to `/v1/chat/completions`.

The user's profile (goals, grade, age, equipment, schedule, discipline) is sent as a structured prompt. The model returns a JSON array representing the full plan, which is parsed and saved to PostgreSQL.

## Implementation

File: `app/src/lib/ai-plan-generator.ts`

```typescript
const url = `${BASE_URL}/v1/chat/completions`;

const res = await fetch(url, {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    "Authorization": `Bearer ${API_KEY}`,
  },
  body: JSON.stringify({
    model: MODEL,
    max_tokens: MAX_TOKENS,
    messages: [
      { role: "system", content: "Output ONLY compact minified JSON array..." },
      { role: "user", content: buildPrompt(input) },
    ],
  }),
});
```

The response text is stripped of any markdown fences (some models wrap JSON in ` ```json `) before `JSON.parse()`.

## Truncated JSON repair

If the model hits the `max_tokens` limit mid-response, the JSON is cut off before the closing `]`. Rather than throwing an error, `repairTruncatedJson` walks the response character-by-character tracking bracket/brace depth and extracts every complete top-level week object, then appends `]` to close the array. This yields a shorter but valid plan (e.g. 2 weeks instead of 4) instead of crashing. The repair runs on any JSON parse failure, not only when `finish_reason === "length"` (OpenRouter sometimes omits that field).

## Model

Default: `anthropic/claude-haiku-4-5` via OpenRouter.

Any OpenAI-compatible model works — change `ANTHROPIC_MODEL` in `.env`. Models that reliably follow JSON-only instructions are preferred.

## Prompt structure

The system prompt instructs the model to act as a JSON API and return a minified JSON array only.

The user prompt provides:
- Discipline (bouldering / sport / trad / ice / alpine) with discipline-specific guidance
- Current and target grade
- Age, plan length (weeks), days per week
- Goals
- Available equipment — with explicit per-equipment rules (e.g. "no hangboard: do NOT include hangboard exercises")

Expected response shape (compact — model is instructed to omit whitespace):
```json
[
  {
    "weekNum": 1,
    "theme": "Foundation & Power Base",
    "days": [
      {
        "dayNum": 1,
        "dayName": "Monday",
        "focus": "Limit bouldering",
        "isRest": false,
        "sessions": [
          {
            "name": "Power Crimp Session",
            "description": "Work V4-V5 problems focusing on powerful movement.",
            "duration": 60,
            "exercises": [
              {
                "name": "Limit Boulder Problems",
                "sets": "5",
                "reps": "3 attempts",
                "rest": "5 min",
                "notes": "Pick problems at your limit (V5-V6). Focus on precise footwork and hip positioning. Stop if fingers feel tweaky."
              }
            ]
          }
        ]
      },
      {
        "dayNum": 2,
        "dayName": "Tuesday",
        "focus": "Rest",
        "isRest": true,
        "sessions": []
      }
    ]
  }
]
```

Field notes:
- `duration` on `DaySession` is `Int` (minutes) in the schema. `actions.ts` coerces string values (e.g. `"45 min"`) to an integer before inserting.
- `sets`, `reps`, `duration`, `rest`, `notes` on `Exercise` are all optional strings in the schema, but the prompt makes `notes` required — every exercise should have a coaching cue.
- Rest days have `isRest: true` and `sessions: []`.
- The prompt requests 3–5 exercises per session with explicit sets/reps/duration/rest values and a 1–2 sentence coaching note per exercise.

## Environment variables

| Variable | Purpose | Example |
|---|---|---|
| `ANTHROPIC_API_KEY` | API key for OpenRouter or direct Anthropic | `sk-or-v1-...` |
| `ANTHROPIC_BASE_URL` | Base URL — code appends `/v1/chat/completions` | `https://openrouter.ai/api` |
| `ANTHROPIC_MODEL` | Model identifier | `anthropic/claude-haiku-4-5` |
| `ANTHROPIC_MAX_TOKENS` | Max output tokens | `5000` |

**OpenRouter base URL:** `https://openrouter.ai/api` — do NOT include `/v1`. The code appends `/v1/chat/completions` itself.

**Direct Anthropic:** set `ANTHROPIC_BASE_URL=https://api.anthropic.com` and model `claude-haiku-4-5` (no `anthropic/` prefix).

## OpenRouter credit

OpenRouter enforces a per-request token cap based on the key's monthly credit limit. If `ANTHROPIC_MAX_TOKENS` exceeds what the key can afford, the API returns `402`. With detailed exercise notes, plans are larger than before — budget ~2000–2500 output tokens for a 4-week plan. Keep the key's monthly limit above `ANTHROPIC_MAX_TOKENS`.

If the plan comes back shorter than requested (e.g. 2 weeks instead of 4), the response was likely truncated and repaired — increase `ANTHROPIC_MAX_TOKENS` or the key's credit limit.

## Error handling

`generatePlanWithAI` throws on:
- Non-200 API response (includes the status and response body)
- Empty `choices[0].message.content`
- JSON parse failure that cannot be repaired by `repairTruncatedJson`
- Empty or non-array parsed result

These errors propagate up through `createPlan` and surface as Next.js server-action errors (page stays on `/onboarding` in the browser). Server-side errors are logged in `docker compose logs web`.
